# هسته موتور حسابداری هوشمند — فاز ۱ (MVP اجراشده و بک‌تست‌شده)

## این چیست؟
پیاده‌سازی واقعی و اجراشدنیِ **هسته دفترداری دوطرفه (Double-Entry Ledger Core)** طبق معماری
Clean Architecture / DDD که در پرامپت خواسته شده بود. چون در این محیط اجرا SDK دات‌نت و
اینترنت در دسترس نبود، هسته با **Python استاندارد (بدون هیچ پکیج خارجی)** نوشته شد تا واقعاً
قابل اجرا و تست باشد. معماری و قوانین کسب‌وکار عیناً همان چیزی است که در نسخه‌ی نهایی
C# .NET 9 / EF Core / PostgreSQL پیاده خواهد شد — فقط لایه‌ی زیرساخت تعویض می‌شود
(اصل Dependency Inversion، دقیقاً طبق درخواست پرامپت).

## ساختار (Clean Architecture)
```
accounting_core/
  domain/           <- قوانین کسب‌وکار خالص (بدون وابستگی به فریم‌ورک)
    entities.py     <- Account, JournalEntry, JournalLine + قانون دوبل
  application/      <- Use Caseها (سرویس‌ها)
    services.py     <- ChartOfAccountsService, JournalService, TrialBalanceService, ClosingService
  infrastructure/   <- پیاده‌سازی دیتابیس (قابل تعویض)
    repositories.py <- SQLite repositories + Audit Log
  tests/
    test_accounting_core.py  <- ۱۳ تست، همه PASS
  demo.py           <- دموی end-to-end با گزارش تراز آزمایشی
```

## چه قوانینی واقعاً پیاده و تست شده‌اند؟
- ✅ سیستم حسابداری دوبل (بدهکار = بستانکار الزامی)
- ✅ رد خودکار سند نامتوازن یا تک‌ردیفی
- ✅ کدینگ شناور حساب‌ها (گروه-کل-معین) + جلوگیری از کد تکراری/والد نامعتبر
- ✅ منع ثبت سند روی حساب‌های غیرقابل‌ثبت (parent accounts)
- ✅ تراز آزمایشی خودکار (چهارستونی) با اثبات ریاضی برابری بدهکار/بستانکار
- ✅ سند برگشتی (Reversal) بدون حذف سند اصلی — طبق استاندارد حسابرسی
- ✅ Audit Trail کامل روی هر عملیات ثبت/برگشت
- ✅ سند اختتامیه خودکار و انتقال سود/زیان خالص به حساب سود انباشته

## اجرا
```bash
cd accounting_core
python3 -m unittest tests.test_accounting_core -v   # نتیجه: Ran 13 tests ... OK
python3 demo.py                                     # دموی end-to-end با گزارش تراز
```

## نقشه راه فازهای بعدی (مطابق ۱۲ بند "خروجی مورد انتظار" پرامپت اصلی)
این پروژه به اندازه یک ERP کامل (حسابداری+انبار+حقوق‌ودستمزد+تولید+CRM+هوش‌مصنوعی+
سامانه مودیان+چندسکویی) است و واقع‌بینانه در چند فاز جدا ساخته می‌شود، نه یک پاسخ:

| فاز | محتوا | خروجی |
|---|---|---|
| **۱ (انجام شد)** | هسته دفترداری دوطرفه، کدینگ شناور، تراز، اختتامیه، Audit | کد+تست PASS ✅ |
| ۲ | خزانه‌داری (صندوق/بانک/چک) + خرید/فروش + انبار (FIFO/LIFO/میانگین) | ماژول‌های Domain جدید روی همین هسته |
| ۳ | حقوق‌ودستمزد + دارایی ثابت + پروژه/مراکز هزینه | ماژول‌های Domain جدید |
| ۴ | لایه API (REST + Swagger) روی FastAPI یا ASP.NET Core | مستقل از هسته دامنه |
| ۵ | مهاجرت زیرساخت از SQLite به PostgreSQL + Multi-Company/Branch/Currency/FiscalYear | فقط لایه Infrastructure عوض می‌شود |
| ۶ | یکپارچگی سامانه مودیان (صورتحساب الکترونیکی، شناسه یکتا/کالا/خدمت، امضای دیجیتال) | ماژول Plugin مجزا |
| ۷ | فرانت‌اند (React+TS برای وب / Avalonia یا WPF برای ویندوز) | مستقل، مصرف‌کننده API |
| ۸ | هوش مصنوعی (پیشنهاد حساب معین، کشف تقلب، پیش‌بینی نقدینگی، OCR فاکتور) | سرویس‌های جدا، مصرف‌کننده API |
| ۹ | امنیت کامل (JWT، RBAC، رمزنگاری، MFA) | Cross-cutting layer |
| ۱۰ | مستندسازی (SRS، ERD، UML، Deployment/User/Developer Guide) | اسناد فنی |
| ۱۱ | تست‌های Integration/UI + بسته نصب Windows + Auto-Update | DevOps |

**پیشنهاد من:** بگویید کدام فاز را بعدی می‌خواهید بسازیم (مثلاً «ماژول انبار با FIFO» یا
«لایه API روی همین هسته») تا با همین سطح از دقت و بک‌تست واقعی جلو برویم.
